Three-layer Flask request pipeline inside this leaf:
- `routes/ecs.py` defines the `/ecs` Blueprint with CRUD + lifecycle endpoints (`/tasks`, `/tasks/<id>/start|stop|status|logs|retry|delete`). It is stateless: it only parses JSON via `CreateECSTaskRequest.from_payload`, delegates to `ECSService`, and serializes results through `ECSTaskResponse.from_model().to_dict()`. Three `@ecs_bp.errorhandler` registrations map custom exceptions to uniform `{"error":{"message":...}}` responses.
- `services/ecs_service.py` owns all business logic. `ECSService` resolves `MigrationJob` / `AWSConnection` rows, calls `AWSClient.assume_role(...)` then constructs per-call boto3 clients for `ecs` (run_task, stop_task, describe_tasks) and `logs` (get_log_events). State transitions are persisted via SQLAlchemy's `db.session`; failures set `ECSTaskStatus.FAILED` before raising `ECSServiceError` subclasses (`ECSTaskNotFoundError`, `ECSValidationError`).
- `services/cloudformation_service.py` is a pure template generator: `CloudFormationService.generate_template` builds an in-memory CloudFormation dict (IAM Role + optional task role) from `AWSConnection` and `CLOUDBRIDGE_AWS_ACCOUNT_ID`, used by other routes to produce downloadable onboarding templates.
- `schemas/ecs.py` holds frozen dataclasses (`CreateECSTaskRequest`, `ECSTaskResponse`) that enforce input validation and serialize model fields to JSON-safe dicts, decoupling DB shape from wire format.
- `models/ecs_task.py` declares the `ECSTask` SQLAlchemy model and the `ECSTaskStatus` value object; list/get endpoints in the route bypass the service and query `ECSTask` directly for read-only operations.
Dependency direction is strictly one-way: routes → services → models/extensions/aws_client; schemas are consumed bidirectionally between routes and services but never import services back.
