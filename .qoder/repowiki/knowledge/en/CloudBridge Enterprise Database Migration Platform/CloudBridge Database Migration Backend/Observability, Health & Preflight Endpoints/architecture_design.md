Two-layer layout per feature: a thin Flask Blueprint in `app/routes/` that parses request payloads and delegates to a stateless service class in `app/services/`. Three independent blueprints are registered at the app root:
- `/health` (no prefix) — single GET `/health` probing SQLAlchemy via `SELECT 1`, returning 200/503.
- `/observability` — CRUD + query endpoints for audit logs (`/audit-log`, `/audit-logs`) plus metrics (`/metrics/migration/<id>`, `/metrics/system`) and outbound CloudWatch push (`/cloudwatch/metric`, `/cloudwatch/log`). Error handlers translate `ObservabilityValidationError` / `ObservabilityServiceError` into `{error:{message}}` JSON.
- `/preflight` — POST `` runs a multi-step readiness check across STS assume-role, region/IAM permissions, Secrets Manager access, and TCP reachability of source/destination databases.
- `/websocket` — plain functions (`handle_connect`, `handle_join_migration`, etc.) wired by Flask-SocketIO; they call `WebSocketService.join_*_room` / `broadcast_*` helpers that emit events into `migration_<id>` or `ecs_task_<id>` rooms.

Dependency direction is strictly routes → services → models/extensions. Services own all I/O: `ObservabilityService` persists `AuditLog` rows and calls `AWSClient` (boto3) for CloudWatch Metrics/Logs; `PreflightService` reuses `database_config_service.test_tcp_connectivity` and `AWSClient.validate_region_access`/`validate_iam_permissions`; `WebSocketService` is a static facade over `flask_socketio.emit/join_room/leave_room` and `app.extensions.socketio`. No route touches a model directly except the preflight route's one-off `AWSConnection.query.get`/`DatabaseConfig.query.get` used for backward-compatible `database_config_id` resolution.
