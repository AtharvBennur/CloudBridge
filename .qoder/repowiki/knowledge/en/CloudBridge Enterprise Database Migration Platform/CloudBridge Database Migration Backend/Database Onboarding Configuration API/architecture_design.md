Four-file Flask MVC split:
- `routes/database_config.py` defines a single Blueprint at `/database-configs` exposing CRUD plus an `/aws-connections/<id>/secrets` sub-resource; every handler is guarded by the shared `@login_required` middleware and registers two custom error handlers (`DatabaseConfigValidationError`→400, `DatabaseConfigNotFoundError`→404).
- `services/database_config_service.py` owns all business logic: request parsing via schema classes, TCP socket reachability probing (`test_tcp_connectivity`, with simulated-host bypass), AWS connection resolution against `app.models.aws_connection.AWSConnection`, and delegation to `SecretManagerService` for create/update/validate/delete of secrets. Two domain exceptions (`DatabaseConfigValidationError`, `DatabaseConfigNotFoundError`) bubble up to route-level handlers.
- `schemas/database_config.py` holds frozen dataclasses (`CreateDatabaseConfigRequest`, `DatabaseConfigResponse`, `DeleteDatabaseConfigResponse`) that normalize payloads in `from_payload` and serialize models via `to_dict`/`from_model`, keeping serialization out of the model layer.
- `models/database_config.py` declares the SQLAlchemy `DatabaseConfig` table (`database_configs`) with a `DatabaseConfigType` enum-like class constraining values to POSTGRESQL/MYSQL/ORACLE/SQL_SERVER.
Dependency direction is strictly routes → service → (schema + model + external services); the service depends on `app.extensions.db` and `app.services.secrets_manager_service.SecretManagerService` but never imports the blueprint.
