Four-file vertical slice following a Route → Service → Model flow:
- `routes/notification.py` defines the `notification_bp` Blueprint under `/notifications`, registers Flask error handlers for the module's custom exceptions (`NotificationValidationError`, `NotificationConfigNotFoundError`, `NotificationDeliveryError`, `NotificationServiceError`), and delegates all business logic to `NotificationService`. Endpoints are gated by the shared `@login_required` middleware.
- `services/notification_service.py` owns delivery orchestration: it queries enabled `NotificationConfig`s subscribed to an event type, persists `Notification` records with PENDING status, then dispatches through `_deliver_notification` which branches on `NotificationType` into `_send_email_notification` (SMTP stub), `_send_slack_notification` (Slack Blocks JSON via `requests.post`), and `_send_webhook_notification` (generic POST with parsed headers/payload). Failed deliveries mark the record FAILED and log via `current_app.logger`; `retry_failed_notifications` walks FAILED rows up to `max_retries`, flipping them RETRYING before re-delivery.
- `models/notification.py` declares SQLAlchemy models `NotificationConfig` and `Notification` plus enum-like classes `NotificationType`, `NotificationEventType`, `NotificationStatus` exposing a `VALUES` set used for validation. JSON fields (`webhook_headers`, `subscribed_events`, `payload`) are stored as TEXT and round-tripped through `json.dumps`/`json.loads`.
- `schemas/notification.py` holds frozen dataclass DTOs (`CreateNotificationConfigRequest`, `SendNotificationRequest`, `NotificationConfigResponse`, `NotificationResponse`) with `from_payload` / `from_model` / `to_dict` helpers that perform request validation and model↔JSON conversion, keeping routes free of parsing logic.
Dependency direction is strictly one-way: routes import schemas + service; service imports models + `app.extensions.db`; schemas import models only for typing. No cross-imports back to routes or services.
